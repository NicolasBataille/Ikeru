# Ikeru - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for Ikeru, decomposing the requirements from the PRD, UX Design, and Architecture into implementable stories.

## Requirements Inventory

### Functional Requirements

FR1: Learner can review due cards using FSRS scheduling algorithm with optimized intervals for 90% retention
FR2: Learner can see new cards introduced according to the kanji knowledge graph (component radicals before composite kanji)
FR3: System can detect leech cards (repeatedly failed items) and flag them for companion intervention
FR4: Learner can view their review queue size, upcoming reviews, and daily review forecast
FR5: System can track response times, error rates, and accuracy per card across all review sessions
FR6: System can log all learning activity and progression data locally, regardless of network state
FR7: Learner can study kanji with radical decomposition (RTK/KKLC hybrid), readings, meanings, and example vocabulary
FR8: Learner can view personalized mnemonics for each kanji, generated by AI and cached locally
FR9: Learner can study vocabulary in context with example sentences sourced from Tatoeba corpus
FR10: Learner can study grammar points aligned to JLPT levels with concise, example-driven explanations
FR11: Learner can practice grammar through fill-in-the-blank exercises (particles, conjugations)
FR12: Learner can read comprehensible input passages graded at i+1 level relative to their current proficiency
FR13: Learner can view stroke order animations for kana and kanji
FR14: Learner can practice stroke order through guided tracing exercises
FR15: Learner can submit handwritten kanji for on-device recognition and feedback
FR16: Learner can practice typed sentence construction exercises
FR17: Learner can listen to audio pronunciation of vocabulary and sentences at variable speeds
FR18: Learner can perform shadowing exercises (listen then repeat) with progressive difficulty
FR19: Learner can study comprehensible input audio passages graded by JLPT level
FR20: Learner can receive on-device pronunciation feedback on spoken Japanese
FR21: System can analyze pitch accent patterns and provide per-pattern accuracy tracking
FR22: Learner can practice isolated word pronunciation with real-time feedback
FR23: Learner can receive pronunciation feedback during shadowing exercises
FR24: Learner can engage in AI-powered conversational practice adapted to their current JLPT level
FR25: System can select the appropriate AI tier transparently (on-device → Gemini → Claude → local GPU)
FR26: Learner can interact with the AI conversation partner via text, voice, or both
FR27: Learner can initiate natural language chat with the companion at any time
FR28: Companion can proactively intervene when leech cards are detected
FR29: Companion can conduct weekly check-in conversations with debate capability
FR30: Companion can modify planner weights and learning data based on check-in outcomes
FR31: Learner can export companion check-in insights for external AI agents
FR32: System can fall back to on-device AI when offline
FR33: System can compose each study session dynamically based on learner state
FR34: System can adapt session content to available time
FR35: System can adapt session content to context (silent mode)
FR36: System can balance all four skills to prevent skill gaps
FR37: System can sequence activities based on pedagogical appropriateness for current level
FR38: Learner can view a progress dashboard with per-skill breakdown and JLPT estimate
FR39: Learner can earn XP from completing learning activities
FR40: Learner can level up based on accumulated XP
FR41: Learner can unlock attributes and rewards through progression
FR42: Learner can earn loot drops from learning sessions
FR43: Learner can unlock lootboxes by completing challenges and tests (infinite retries)
FR44: System can map RPG progression to real skill growth
FR45: New user can enter their name at first launch and receive a guided tour
FR46: Learner can change their display name at any time, reflected everywhere
FR47: System can maintain multiple independent local profiles
FR48: Learner can switch between profiles in-app
FR49: Onboarding tour can explain the journey, introduce companion, present first session
FR50: Learner can perform kana/kanji recognition quizzes (4-choice) on Watch
FR51: Learner can perform audio pronunciation drills via Watch speaker/mic
FR52: Learner can receive haptic-guided pitch accent feedback on Watch
FR53: Learner can view progress via Watch complications
FR54: System can sync SRS state, progress, and RPG data between Watch and iPhone in real-time
FR55: Watch sessions completed offline can queue and sync when connection restores
FR56: Learner can manually trigger iCloud backup from Settings
FR57: Learner can restore learning data from iCloud backup via Settings
FR58: Learner can export all learning data in JSON, CSV, and Parquet formats
FR59: System can display Live Activities (session progress, next review countdown)
FR60: System can display flashcards in StandBy mode
FR61: Learner can trigger study sessions via Siri Shortcuts
FR62: System can appear in Spotlight suggestions
FR63: System can send configurable push notifications
FR64: System can progressively load content by JLPT level
FR65: System can display an attribution screen

### NonFunctional Requirements

NFR1: SRS card transitions under 100ms
NFR2: FSRS scheduling sub-millisecond per card
NFR3: Pronunciation feedback within 500ms
NFR4: Watch interactions within 200ms
NFR5: App launch to first interaction under 2 seconds
NFR6: AI conversation — on-device 1-2s, cloud up to 5s with indicator
NFR7: Session composition under 500ms
NFR8: Zero tolerance for SRS data corruption — atomic mutations
NFR9: Offline-first guarantee — core loop identical with/without network
NFR10: Watch sync resilience — no data loss on connection drop
NFR11: Profile isolation — corruption cannot cross profiles
NFR12: iCloud backup integrity — all-or-nothing restore
NFR13: Crash recovery — resume exactly where learner left off
NFR14: AI tier switching invisible to learner
NFR15: AI tier fallback within 2 seconds
NFR16: WatchConnectivity bidirectional real-time sync
NFR17: iCloud manual backup with graceful quota/conflict handling
NFR18: iOS system features follow latest API contracts
NFR19: Content pipeline loading seamless — no blocking during sessions

- AR5: .xcconfig files for build-time secrets excluded via .gitignore

- AR7: SwiftLint configuration for code consistency
- AR8: App Group entitlement for Widget/Live Activity shared data access
- AR9: Bonjour/mDNS discovery for local RTX 5090 GPU server
- AR10: FSRS algorithm Swift implementation (port from open-spaced-repetition or native)
- AR11: Kanji knowledge graph as DAG with topological sort for ordering
- AR12: AIProvider protocol with 4 implementations (FoundationModels, Gemini, Claude, LocalGPU)
- AR13: Content attribution screen crediting CC-licensed sources (KanjiVG, Tatoeba, KANJIDIC, RADKFILE)

## Epic 1: Foundation & First Learning Session

The learner can open the app, create a profile, complete an onboarding tour, and do their first kana SRS review session with basic XP reward.

### Story 1.1: Project Scaffolding and Design System Foundation

As a developer,
I want the Xcode project set up with all targets, IkeruCore package, CI/CD, and IkeruTheme design tokens,
So that all subsequent development has a consistent foundation to build on.

**Acceptance Criteria:**

**Given** a fresh checkout of the repository
**When** the project is opened in Xcode
**Then** iOS app target, watchOS target, Widget Extension target, and IkeruCore SPM package are configured and build successfully
**And** GitHub Actions CI pipeline runs lint (SwiftLint) + build on push
**And** IkeruTheme enum defines all design tokens (colors, typography, spacing, radius, animation, shadows)
**And** IkeruCard, IkeruButtonStyle (.primary/.secondary/.rpg/.danger/.ghost), and toast system are implemented in IkeruCore
**And** NavigationCoordinator with path-based navigation is implemented
**And** Tab bar with 5 tabs (Home, Study, Companion, RPG, Settings) is visible
**And** Noto Serif JP (Bold + Medium) is bundled
**And** Dark mode is enforced (no light mode)
**And** .xcconfig for secrets is in .gitignore
**And** skeleton shimmer loading pattern is available

### Story 1.2: SwiftData Models and FSRS Engine

As a learner,
I want my learning progress tracked reliably with an intelligent scheduling algorithm,
So that I review cards at optimal intervals for 90% retention.

**Acceptance Criteria:**

**Given** the app is launched
**When** SwiftData models are initialized
**Then** UserProfile, Card (with FSRSState, CardType enum, Grade enum), and ReviewLog models exist
**And** FSRS scheduling algorithm computes next review date in sub-millisecond per card
**And** Card state mutations are atomic — no partial writes on crash
**And** All learning data is logged locally regardless of network state
**And** Response times and accuracy are tracked per card per review

### Story 1.3: Profile Creation and Onboarding Tour

As a new learner,
I want to enter my name and take a guided tour of the app,
So that I understand how Ikeru works before my first session.

**Acceptance Criteria:**

**Given** the app is launched for the first time
**When** no profile exists
**Then** a name entry screen appears with a single text field
**And** after entering a name, a 3-screen guided tour plays (journey overview, companion introduction, first session preview)
**And** the tour uses Lottie animations
**And** after the tour, the learner is guided to start their first session
**And** the profile name is reflected across all UI surfaces

### Story 1.4: SRS Card Review with Swipe Grading

As a learner,
I want to review flashcards by swiping to grade them,
So that I can quickly process my review queue with muscle memory.

**Acceptance Criteria:**

**Given** the learner has cards due for review
**When** a card is presented
**Then** the card displays on an SRSCardView with card stack depth (next card peeking behind)
**And** swiping left grades "Again", right grades "Good", up grades "Easy", down grades "Hard"
**And** grade buttons are also visible at the bottom as fallback
**And** card transitions complete in under 100ms
**And** correct answers show jade green flash + `.notification(.success)` haptic
**And** wrong answers show vermillion flash + `.notification(.warning)` haptic
**And** FSRS algorithm updates the card's next review date based on grade

### Story 1.5: Basic Session Composition and First Session

As a learner,
I want the app to compose my first study session automatically,
So that I start learning kana immediately without deciding what to study.

**Acceptance Criteria:**

**Given** a new learner who just completed onboarding
**When** they tap "Start Session"
**Then** the planner composes a beginner-appropriate session (hiragana recognition drills)
**And** the session starts in full-screen immersive mode (tab bar hidden)
**And** session progress bar appears at the top
**And** exercises are served sequentially
**And** the session can be paused by swiping down
**And** on session completion, a summary screen shows cards reviewed and XP earned
**And** the learner returns to the home screen

### Story 1.6: Basic XP and Level-Up

As a learner,
I want to earn XP for completing exercises and level up,
So that I feel a sense of progression from my very first session.

**Acceptance Criteria:**

**Given** the learner completes a learning activity (card review, exercise)
**When** XP is awarded
**Then** "+XP" text floats up and fades with `.impact(.light)` haptic
**And** XPBarView (compact variant) fills incrementally during session
**And** when XP threshold is crossed, level-up is triggered
**And** level-up shows a celebration animation with dramatic haptic
**And** RPGState model persists XP and level in SwiftData
**And** home screen displays current level and XP bar

### Story 1.7: Home Screen — "Your World"

As a learner,
I want to see my learning status and RPG progress when I open the app,
So that I know where I stand and what's next.

**Acceptance Criteria:**

**Given** the learner opens the app (not first launch)
**When** the home screen loads
**Then** the screen shows: level + XP bar at top, learning summary (cards due, basic progress), and "Start Session" CTA with session preview
**And** app launch to first interaction completes in under 2 seconds
**And** the home screen displays even when offline

## Epic 2: Kanji & Reading System

The learner can study kanji with radical decomposition, mnemonics, vocabulary in context, grammar points with exercises, and comprehensible input passages.

### Story 2.1: Kanji Knowledge Graph and Content Loading

As a learner,
I want kanji to be organized so I always learn radicals before the kanji that use them,
So that decomposition always makes sense when I encounter a new kanji.

**Acceptance Criteria:**

**Given** the app has static content bundles
**When** N5 content is loaded
**Then** pre-built SQLite bundles for KanjiVG, KANJIDIC, RADKFILE, and Tatoeba are accessible via ContentRepository
**And** kanji knowledge graph (DAG) is built with radical → kanji edges
**And** topological sort guarantees radicals are introduced before their composite kanji
**And** content loads progressively (N5 at install, higher levels when approaching)
**And** content loading never blocks a session

### Story 2.2: Kanji Study with Radical Decomposition

As a learner,
I want to study kanji by seeing their radical components, readings, meanings, and vocabulary,
So that I understand how kanji are constructed and can remember them.

**Acceptance Criteria:**

**Given** a new kanji is introduced in a session
**When** the kanji study view is presented
**Then** KanjiDisplayView shows the kanji in Noto Serif JP (64pt, warm white on dark)
**And** radical decomposition is displayed with each component radical and its meaning
**And** on'yomi and kun'yomi readings are shown
**And** example vocabulary using this kanji is displayed with sentences
**And** tapping the kanji reveals stroke order animation
**And** progressive disclosure: tap sections to expand/collapse

### Story 2.3: AI Mnemonic Generation and Caching

As a learner,
I want personalized mnemonics for each kanji,
So that I can remember them using associations that make sense to me.

**Acceptance Criteria:**

**Given** a kanji is displayed in study view
**When** the mnemonic section is shown
**Then** a personalized mnemonic is displayed, generated by AI using the learner's context
**And** generated mnemonics are cached in SwiftData per card
**And** the learner can tap to regenerate a mnemonic if the current one doesn't resonate
**And** mnemonic generation works offline via on-device FoundationModels (lower quality acceptable)

### Story 2.4: Vocabulary and Grammar Exercises

As a learner,
I want to study vocabulary in context and practice grammar through exercises,
So that I build reading comprehension alongside kanji knowledge.

**Acceptance Criteria:**

**Given** the planner includes vocabulary or grammar in a session
**When** a vocabulary item is presented
**Then** the word, reading, meaning, and example sentences (from Tatoeba) are displayed
**And** grammar points show concise explanations with examples, aligned to JLPT level
**And** fill-in-the-blank exercises test particle usage and conjugation
**And** instant feedback on answers (jade/vermillion flash + haptic)

### Story 2.5: Comprehensible Input Reading Passages

As a learner,
I want to read graded passages at my level,
So that I practice reading comprehension with material that's challenging but not overwhelming.

**Acceptance Criteria:**

**Given** the learner's current proficiency level
**When** a reading passage is presented
**Then** the passage is graded at i+1 (slightly above current level)
**And** kanji in the passage that the learner knows are shown without furigana
**And** unknown kanji show furigana above
**And** tapping a word shows its definition

## Epic 4: Listening & Speaking

The learner can practice listening comprehension, shadowing, and pronunciation with real-time pitch accent analysis and feedback.

### Story 4.1: Audio Playback and Listening Exercises

As a learner,
I want to hear Japanese pronunciation at variable speeds,
So that I train my ear to recognize sounds and words.

**Acceptance Criteria:**

**Given** a listening exercise is presented
**When** audio plays
**Then** vocabulary and sentence audio plays via TTS or cached audio
**And** playback speed is adjustable (0.5x, 0.75x, 1.0x, 1.25x)

### Story 4.2: Shadowing Exercises

As a learner,
I want to listen to Japanese audio and repeat it back,
So that I improve my pronunciation and speech rhythm.

**Acceptance Criteria:**

**Given** a shadowing exercise is presented
**When** audio plays and the learner repeats
**Then** the app records the learner's speech via microphone
**And** pronunciation feedback is provided comparing to the target
**And** progressive difficulty from simple phrases to natural-speed conversation
**And** exercises available for individual words and full sentences

### Story 4.3: Pitch Accent Analysis and Tracking

As a learner,
I want real-time feedback on my pitch accent patterns,
So that I develop natural-sounding Japanese pronunciation from the start.

**Acceptance Criteria:**

**Given** the learner speaks a Japanese word
**When** pronunciation is analyzed
**Then** on-device F0 contour analysis identifies the pitch accent pattern (頭高, 中高, 尾高, 平板)
**And** feedback shows whether the pattern matches the target
**And** per-pattern accuracy is tracked over time
**And** pronunciation feedback returns within 500ms
**And** pitch accent tracking works for isolated words and during shadowing

## Epic 5: Adaptive Study Planner & Dashboard

The app intelligently composes mixed-skill sessions adapting to available time, context, and skill balance. The learner sees detailed progress.

### Story 5.1: Full Adaptive Session Composition

As a learner,
I want the app to compose sessions that balance all four skills based on my current state,
So that I progress evenly without developing skill gaps.

**Acceptance Criteria:**

**Given** the learner taps "Start Session"
**When** the planner composes the session
**Then** session composition considers: error rates, response times, skill balance ratios, SRS queue, and time available
**And** all four skills (reading, writing, listening, speaking) are balanced over time
**And** activities are sequenced by pedagogical appropriateness for current level
**And** session composition completes in under 500ms

### Story 5.2: Immersive Session Mode with Exercise Transitions

As a learner,
I want sessions to flow smoothly between different exercise types,
So that I stay in a state of flow without jarring transitions.

**Acceptance Criteria:**

**Given** an active session with mixed exercise types
**When** one exercise completes and the next begins
**Then** transitions use matchedGeometryEffect for smooth card morphing
**And** SessionProgressBar shows current position, exercise type icons, and elapsed time
**And** full-screen immersive mode hides the tab bar
**And** the session is pausable via swipe-down gesture

### Story 5.3: Time and Context Adaptation

As a learner,
I want sessions to adapt to how much time I have and whether I can use audio,
So that I can study effectively in any context.

**Acceptance Criteria:**

**Given** the learner starts a session
**When** available time is selected or detected
**Then** micro-sessions (2-5 min) contain only SRS reviews
**And** focused sessions (30 min) contain mixed exercises across all skills
**And** when system volume is muted, audio exercises are automatically excluded
**And** session preview shows estimated time and exercise breakdown

### Story 5.4: Progress Dashboard

As a learner,
I want to see my detailed progress across all skills,
So that I understand my strengths, weaknesses, and JLPT trajectory.

**Acceptance Criteria:**

**Given** the learner navigates to the dashboard
**When** progress data is displayed
**Then** SkillRadarView shows 4-axis balance (reading/writing/listening/speaking)
**And** JLPT level estimate is shown (e.g., "N5: 42% mastered")
**And** review queue size, upcoming reviews, and daily forecast are visible
**And** monthly progress snapshots are available for trend analysis

## Epic 6: AI Companion & Conversation Partner

The learner has an AI companion for conversational practice, natural language help, leech intervention, and weekly check-ins with debate capability.

### Story 6.1: AI Tier Router with Provider Protocol

As a learner,
I want AI features to work seamlessly regardless of my network connection,
So that I always have the best available AI assistance.

**Acceptance Criteria:**

**Given** an AI request is made
**When** the AIRouter evaluates available tiers
**Then** it selects the appropriate tier: on-device (offline/simple) → Gemini (online/medium) → Claude (complex) → LocalGPU (batch)
**And** tier switching is invisible to the learner
**And** on tier failure, fallback to next available tier completes within 2 seconds
**And** API keys are stored in iOS Keychain (never in source code)
**And** local GPU server is discovered via Bonjour/mDNS

### Story 6.2: AI Conversation Partner

As a learner,
I want to practice Japanese conversation with an AI partner at my level,
So that I develop speaking and comprehension skills through realistic dialogue.

**Acceptance Criteria:**
